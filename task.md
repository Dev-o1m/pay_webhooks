# Тестовое задание: Платёжный шлюз (с интеграцией провайдера)

Необходимо реализовать упрощённый платёжный шлюз, который:

- принимает платёжные заявки (выплаты) от мерчантов
- проверяет доступный баланс мерчанта перед созданием заявки
- при создании заявки отправляет запрос на обработку внешнему провайдеру
- принимает и обрабатывает вебхуки (уведомления) от провайдера об изменении статуса заявки
- обновляет статус заявки и списывает средства с баланса мерчанта только при успешном статусе от провайдера

### Важно
- Несколько мерчантов могут отправлять запросы конкуррентно и с высокой частотой (>5 запросов/сек)
- У мерчанта должна быть возможность получить информацию о своем профиле и балансе (одна GET ручка)
- Реального провайдера подключать не нужно.    
- Ваша задача — реализовать обе стороны взаимодействия: отправку запроса провайдеру и приём + обработку его ответов через вебхук.
- Проект должен быть покрыт тестами, .env для конфигурации запуска тестов должен быть в репозитории
- .env для запуска проекта (для оценки) должен находится в репозитории
- Проект должен запускаться следующей последовательностью команд:
  - git clone \{Ваш репозиторий\}
  - docker compose up --build
- Первый запуск проекта должен запускать миграцию, которая добавит в БД необходимые для проверки задания данные (пользователи, балансы)
- Инструкция по деплою и запуску тестов должна быть в README.md в корне проекта

### Основные правила
- Создать заявку можно только если сумма ≤ доступного баланса
- После успешного ответа от провайдера (через вебхук) — списать сумму с баланса
- При отмене / ошибке от провайдера — баланс не трогается
- Статусы заявки: Создана → В обработке → Успех / Отмена

### Технические требования
Стек:
- Python 3.12
- FastAPI (асинхронный)
- SQLAlchemy 2.0 + Alembic
- PostgreSQL
- Redis
- Docker + docker-compose
- pytest

Сущности (минимально):
- Merchant / User
- Balance
- Payment / Order

### Аутентификация
- API-токен в заголовке + подпись тела запроса (любой разумный механизм)

### Дополнительно
- Перед отправкой заявки провайдеру добавить искусственную задержку 1-2 секунды (имитация отработки внутренней логики)
- Реализовать обработку входящих вебхуков от провайдера
- Мерчант, провайдер - другой платежный шлюз

### API-документация провайдера

#### Создание платежа

POST /api/v1/payments

Тело запроса:
{
	"external_invoice_id": "string",
	"amount": "string", // Формат: "10.00"
	"callback_url": "string"
}

Ответ:

201 CREATED
{
	"id": "string",
	"external_invoice_id": "string",
	"amount": int,
	"callback_url": "string",
	"status": "Created"
}

#### Вебхук

POST {callback_url}

Тело вебхука:
{
	"id": "string",
	"external_invoice_id": "string",
	"status": "string"
}

Статусы: Created, Completed, Canceled
